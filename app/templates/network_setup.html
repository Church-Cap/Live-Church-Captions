Network setup for audience captions

Church Cap serves captions from this computer to phones, displays, and OBS over your local network. The detected audience URL is {{ base_url }}. For regular services, use either a stable local hostname or a reserved IP address so printed QR codes and saved OBS sources do not change unexpectedly.

Recommended local network layout

Keep Church Cap local to the building network. Audience devices only need access to the public viewer port; operator controls should stay password-protected and limited to trusted devices.

Audience phones → Church Cap viewer port 8080
Room display / OBS → Church Cap display or OBS URL
Operator dashboard → local/trusted access on port 9090
Internet port forwarding → off

If you are using the Church Cap Appliance, keep the operator side on the secure local port 9090. Audience devices should use the QR code shown in the operator page.

Hostname or reserved IP

A local hostname is easier to read, while a reserved IP address can be more reliable on networks that block Bonjour/mDNS.

Preferred hostname example:  http://church-cap.local:8080
Reserved IP example:        http://192.168.1.50:8080

On macOS, the setup script can offer to set church-cap.local. On Windows and Linux, use your router's DHCP reservation or local DNS tools. If a phone cannot open a .local link, use the direct IP QR code shown in the operator page.

To pin the audience link manually, set this in .env:

PUBLIC_BASE_URL=http://church-cap.local:8080

Guest Wi-Fi rules

For guest or audience Wi-Fi, allow access to the caption computer and block unnecessary access to the rest of the local network.

Allow: audience network → caption computer port 8080
Optional: trusted AV devices → display/OBS/operator ports
Block: audience network → other LAN devices
Protect: operator page with a strong password

Exact firewall wording varies by router. The principle is simple: viewers should reach captions, not the whole office or AV network.

HTTPS options

Church Cap can run over local HTTP for many offline church networks. HTTPS is stronger for operator and service-leader controls, but local certificates require planning because browsers will not silently trust a certificate generated by the caption computer.

Managed/test devices: mkcert or an installed local CA
No-warning deployment: real domain + DNS-01 certificate
Fallback: local HTTP on a trusted isolated network

Self-signed certificates will show browser warnings unless each device trusts the issuing certificate authority. For details, see the HTTPS guidance in the project documentation.
