# v0.7.3 Implementation Status

Version: v0.7.3

v0.7.3 is a focused end-of-speech transcription reliability release on the v0.7.2 updater-recovery baseline. Testing showed that a rolling-window decode could finish after the speaker paused and replace the last speech-backed wording with plausible text inferred from silence or background noise. The fix rejects that publication path without adding another model pass or a fixed caption delay.

## v0.7.3 End-Of-Speech Guard

| Issue | v0.7.3 implementation | Latency and regression boundary |
| --- | --- | --- |
| A rolling buffer still contains old speech after the speaker pauses | Each scheduled pass records whether voice was recent when its audio snapshot was captured. A non-empty result from a pass that began outside that grace period is treated like an empty result, so it cannot replace the pending speech-backed wording. | Uses the existing inference and cadence; no extra model call, browser hold, or fixed wait is added. A new utterance that begins during an older inference is captured on the next normally scheduled pass. |
| Faster Whisper metadata was not checked after decoding | Both backends now share the established no-speech, average-log-probability, repeated-output, and pathological-character checks. | Thresholds match the standard Whisper path that already shipped; no global RMS or VAD threshold was raised, protecting quieter speakers. |
| Faster Whisper exposed purpose-built VAD and silence controls that were too conservative for this live endpoint | Both backends use word timestamps, `hallucination_silence_threshold=0.5`, and streaming-oriented Silero VAD (`160 ms` minimum silence and `100 ms` speech padding). Standard Whisper reuses the Silero model already installed with Faster Whisper. | Faster Whisper replaces its existing VAD configuration; standard Whisper adds only the lightweight VAD classifier, not another transcription pass. |
| A pass inside the voice-grace period could still contain a growing silence tail | Both backends bound each existing inference input at the last voiced callback plus a 350 ms endpoint-analysis tail. Silero removes post-speech audio before decoding while retaining 100 ms of natural acoustic padding. | Active-speech windows remain unchanged. No fixed publication hold is added. |
| The scheduled loop could treat a second decode of identical post-speech audio as confirmation | A weak word at the live edge is confirmed only when a matching later hypothesis arrives **and the detected speech boundary has advanced by at least 60 ms**. Generated punctuation is not treated as acoustic evidence. | Prevents a pause-time suffix from validating itself without delaying safe, high-confidence words. |
| The diagnostics could not show whether a guard intervened | Service-metrics schema 11 adds a named guard strategy, counts for metadata-rejected and post-speech hypotheses, a total suppressed-word count, speech-bounded pass count, and total trailing silence trimmed. | Reports keep numbers only. No rejected words, captions, confidence values, timestamps, audio, or translations are retained. Schema-9 and schema-10 histories remain readable. |

## v0.7.2 Updater Changes Retained

| Issue | retained v0.7.2 implementation | Regression boundary |
| --- | --- | --- |
| v0.7.0 could download a newer release but could not validate its version | `Settings.app_version` deliberately retains a literal `0.7.3` value alongside the canonical `APP_VERSION`. This is the exact legacy field read by the published v0.7.0 shell updater. An automated regression test requires both values to match. | The settings validator remains code-owned, so a stale `.env` cannot change the running release version. |
| The web UI could display an updating overlay after the updater had already failed | The server retains the detached updater process, polls its exit code through a local authenticated status endpoint, and returns a safe failure message plus the existing `logs/update.log` label. | No log contents or absolute filesystem paths are exposed by the status endpoint. The redacted log remains available through the existing confirmed Diagnostics export. |
| Operators had no recovery action when an update stopped | The Church Cap themed update overlay changes to an **Update stopped** state with **Download diagnostics** and **Return to Updates** actions. Controls are re-enabled after failure. | Successful updates still reconnect through `/health` and reload the operator page when the selected version starts. |
| A release could change between the app check and script download | The selected version is passed to macOS/Linux and Windows scripts through `CHURCH_CAP_UPDATE_TARGET_VERSION`; the script downloads and validates that exact GitHub tag. | Command-line updates without a selected target still use GitHub's latest published release. |
| Repeated update attempts could retain parent log descriptors | The launcher closes its parent copy of the update log handle immediately after the child process starts. | The detached updater keeps its inherited output handle and continues writing to `logs/update.log`. |
| Two web updates could be started concurrently | The update start route returns HTTP 409 while a tracked updater is starting or running. | A failed or completed attempt can be retried after its state is observed. |

The v0.7.2 web-updater recovery path and the v0.7.1 language-settings save action, Service Leader explanation, version-aware translation copy, clearer operator icons, environment-template cleanup, and cross-platform updater readers remain present.

Release verification covers 206 passing automated tests, Python and JavaScript syntax, all shipped shell scripts, authenticated route/render smoke tests, local documentation links, updater compatibility simulation, dependency review, and a rendered action-plan check. The optional Argos set retains the two upstream-pinned advisories documented in `docs/legal/THIRD_PARTY_NOTICES.md`; Church Cap disables its unused Stanza model pipeline and requires trusted model packages.

## v0.7.0 Baseline Retained

The following v0.7.0 action-plan implementation remains present and is regression-tested in v0.7.3.

| Action-plan requirement | Implementation | Evidence |
| --- | --- | --- |
| Preserve low-latency English | Faster-Whisper requests per-word alignment and confidence. Safe words publish on the first pass; only a weak final word within 0.32 seconds of the captured edge waits for a matching decode. Recognition passes use start-to-start cadence, so decode time is subtracted from the configured interval. | Focused edge-guard and cadence tests plus the existing cue/viewer regression suite. Linux measurements remain required. |
| Give rolling speech stable identity | `SourceUnitBuilder` is a word-timestamp-aware local-agreement cue ledger. Consecutive hypotheses revise one `cue_id`; words agreed across passes form an explicit stable prefix and only the newest guarded tail remains mutable. Real word times replace equal segment interpolation where Faster-Whisper supplies them. | Unit tests cover word-aligned corrections, stable-prefix growth, timestamp boundaries, final trim acknowledgement, Whisper final, reset, and private metadata exclusion. |
| Bound rolling-window work | Sealing a cue exposes its immutable audio end to Faster-Whisper. Later passes retain one second of overlap but omit older sealed audio. | Timestamp-boundary tests and Linux transcription-compute comparison. |
| Coalesce revisions, not completed thoughts | Every older queued revision for the same cue is replaced, including an obsolete final revision. Unrelated sealed cues remain durable. | Scheduler tests cover queued and in-flight draft/final supersession by cue ID. |
| Preserve completed speech | Final jobs are never removed for ordinary backpressure. A final-only queue may exceed its normal capacity. | Durability and overload tests. |
| Avoid language starvation | Independent per-language queues are selected round-robin. | Fairness test with uneven Farsi/Chinese demand. |
| Bound overload | Normal capacity is eight jobs per language. Oldest drafts are dropped first and new drafts can be rejected while a final-only queue drains. | Backpressure and recovery tests. |
| Preserve privacy controls | Sensitive mode and Clear reset source-unit context, queues, revision state, and in-flight job generation. | Reset test and existing sensitive-mode test suite. |
| Make behavior measurable | Service-metrics schema 11 retains cue-engine v5 measurements and uses the text-free `silero_endpoint_progress_publication_guard_v3` identity, suppressed-hypothesis counts, suppressed-word total, speech-bounded pass count, and trailing-silence total. | Metrics, schema-9/schema-10 migration, privacy, cadence, responsive timing, drain, and timeout tests plus anonymised service report. |
| Keep the universal viewer stable | The audience shell is viewport-bounded; Live and Session transcript scroll independently in English, translated languages, LTR, and RTL. Live and transcript entries use `cue_id` and reject older revisions, so corrections update the existing cue. A persistent line ledger preserves complete stable-prefix lines, rewraps only the affected tail using measured font width, and reuses line identities when the cue seals. The phone reader has no transform-based line glide; only the subtle newest-word fade remains. Session transcript starts closed, and closing notices grows the panels in place. | Broadcast/client cue-revision tests, a timed correction/extension/seal browser sequence, and responsive browser checks at 390×844, 375×667, and 844×390. |
| Separate Chinese scripts | The audience catalogue exposes `zh-Hans` and `zh-Hant`. Existing providers translate through their Chinese target and OpenCC enforces `t2s` or Hong Kong `s2hk` output. Legacy `zh` migrates to `zh-Hans`. | Provider-routing, locale, real OpenCC profile, and catalogue tests. |
| Add context without waiting for paragraphs | Responsive Context translates the current cue once at least three English words are stable. It normally retranslates after at least 1.5 seconds and three more stable words, while queue coalescing removes obsolete revisions. The completed English wording refines the same translated cue. | Unit and broadcast tests prove unstable tails are excluded, a draft translates before final, cue identity is retained, and the final full cue remains eligible. Linux/native-reader validation remains required. |
| Retire unusably slow context batching | The operator no longer offers two-thought Contextual or four-thought Extended modes. Existing saved values migrate to Responsive Context in both runtime configuration and the live hub. | Configuration, UI-routing, and legacy-mode migration tests. |
| Prevent unexplained storage growth | Diagnostics now calculates application/data/model-cache storage on demand. Logs rotate at 5 MB with two backups, HTTP access logging is disabled, benchmark samples are memory-capped, and cleanup is allow-listed and operator-confirmed. | Storage, active-model protection, cleanup, rotation, and bounded-tail tests pass. |

## Required Linux Validation

Run A/A2 on this v0.7.3 release candidate using the same recording, audio route, Faster-Whisper model, device, and timing preset. If a historical comparison is needed, use the published v0.6.0 release. If English correction frequency and cadence pass, run Responsive Context, Live, and More Stable with Farsi, `zh-Hant`, and both languages on equivalent input at least three times.

Keep Responsive Context as the recommendation only when:

- English operational response has no material regression.
- `source_units.engine` is `word_timestamp_local_agreement_v5`, `interim_strategy` is `immediate_stable_prefix_guarded_edge_tail`, processing p95 is at most 0.010 seconds, and maximum is below 0.025 seconds.
- Faster-Whisper runs report `word_timestamp_passes` and `aligned_words`; the report must retain no words, times, or confidence values.
- `transcription_streaming.guard_strategy` is `silero_endpoint_progress_publication_guard_v3`. During a deliberate speak-then-pause test, `speech_bounded_audio_passes` and `trailing_silence_trimmed_seconds` show input trimming. Weak live-edge suffixes appear only as text-free `edge_words_withheld`; unchanged post-speech audio cannot increase `edge_words_confirmed`. Metadata or post-grace rejections remain numeric only; no caption wording is exported.
- After a clean pause, no new unspoken words appear and the last legitimate caption remains available for normal finalisation. Repeat with a quiet speaker and normal room noise to ensure true final words are not lost.
- When transcription compute p95 remains below the configured refresh interval, start-to-start `pass_interval` p95 should remain close to that interval rather than interval plus compute time.
- Edge-word withholding materially reduces visible last-word reversals without a material first-caption regression.
- First drafts, corrections, and extensions appear on their first recognition pass; there is no server revision hold or browser partial throttle.
- Cue lifetime p95 is materially lower than the previous run and `maximum_revision` no longer reaches long-lived values such as 54 during normal speech.
- Corrected A2 phrases revise one cue; legitimate spoken repetition remains visible.
- Every active language receives completed units in order.
- `cue_first_translation_publish` p50 is at most 3 seconds and p95 is at most 5 seconds on the Linux reference appliance, or any exception is explained by provider/model load.
- Translated draft revisions update one cue in place; completed translated cues do not change or reappear as duplicates.
- `source_units.final_units` is non-zero and boundary reasons look plausible.
- Any `final_jobs_superseded` event is explained by a newer revision of the same cue and the stale result is not published.
- Draft coalescing/backpressure explains overload without an accumulating delayed backlog.
- Each degraded event recovers after load falls.
- Native Farsi and Chinese reviewers confirm no new completeness regression.

The numeric report demonstrates timing and scheduler behavior. Human review remains required for meaning, naturalness, church terminology, script consistency, and reading comfort.
