# v0.7.0 Implementation Status

Version: v0.7.0

This file maps the public v0.7.0 code to the action plan agreed after the v0.6.1 measurements and subsequent Mac and Linux validation. The release combines stable cue identity and correction-in-place, word-timestamp-aware rolling recognition, fair bounded translation queues, responsive context from the stable English live feed, separate Simplified and Hong Kong Traditional Chinese choices, bounded diagnostics storage, and a consistent scrolling viewer.

## Implemented

| Action-plan requirement | Implementation | Evidence |
| --- | --- | --- |
| Preserve low-latency English | Faster-Whisper requests per-word alignment and confidence. Safe words publish on the first pass; only a weak final word within 0.32 seconds of the captured edge waits for a matching decode. Recognition passes use start-to-start cadence, so decode time is subtracted from the configured interval. | Focused edge-guard and cadence tests plus the existing cue/viewer regression suite. Linux measurements remain required. |
| Give rolling speech stable identity | `SourceUnitBuilder` is a word-timestamp-aware local-agreement cue ledger. Consecutive hypotheses revise one `cue_id`; words agreed across passes form an explicit stable prefix and only the newest guarded tail remains mutable. Real word times replace equal segment interpolation where Faster-Whisper supplies them. | Unit tests cover word-aligned corrections, stable-prefix growth, timestamp boundaries, final trim acknowledgement, Whisper final, reset, and private metadata exclusion. |
| Bound rolling-window work | Sealing a cue exposes its immutable audio end to Faster-Whisper. Later passes retain one second of overlap but omit older sealed audio. | Timestamp-boundary tests and Linux transcription-compute comparison. |
| Coalesce revisions, not completed thoughts | Every older queued revision for the same cue is replaced, including an obsolete final revision. Unrelated sealed cues remain durable. | Scheduler tests cover queued and in-flight draft/final supersession by cue ID. |
| Preserve completed speech | Final jobs are never removed for ordinary backpressure. A final-only queue may exceed its normal capacity. | Durability and overload tests. |
| Avoid language starvation | Independent per-language queues are selected round-robin. | Fairness test with uneven Farsi/Chinese demand. |
| Bound overload | Normal capacity is eight jobs per language. Oldest drafts are dropped first and new drafts can be rejected while a final-only queue drains. | Backpressure and recovery tests. |
| Preserve privacy controls | Sensitive mode and Clear reset source-unit context, queues, revision state, and in-flight job generation. | Reset test and existing sensitive-mode test suite. |
| Make behavior measurable | Service-metrics schema 9 identifies cue-engine v5 and its immediate stable-prefix/guarded-edge-tail strategy. It adds text-free word-timestamp passes, aligned-word totals, edge-word withholding/confirmation, actual pass-interval distributions, translated draft/final publications, and time from first English cue update to first translated publication. | Metrics, privacy, cadence, responsive timing, drain, and timeout tests plus anonymised service report. |
| Keep the universal viewer stable | The audience shell is viewport-bounded; Live and Session transcript scroll independently in English, translated languages, LTR, and RTL. Live and transcript entries use `cue_id` and reject older revisions, so corrections update the existing cue. A persistent line ledger preserves complete stable-prefix lines, rewraps only the affected tail using measured font width, and reuses line identities when the cue seals. The phone reader has no transform-based line glide; only the subtle newest-word fade remains. Session transcript starts closed, and closing notices grows the panels in place. | Broadcast/client cue-revision tests, a timed correction/extension/seal browser sequence, and responsive browser checks at 390×844, 375×667, and 844×390. |
| Separate Chinese scripts | The audience catalogue exposes `zh-Hans` and `zh-Hant`. Existing providers translate through their Chinese target and OpenCC enforces `t2s` or Hong Kong `s2hk` output. Legacy `zh` migrates to `zh-Hans`. | Provider-routing, locale, real OpenCC profile, and catalogue tests. |
| Add context without waiting for paragraphs | Responsive Context translates the current cue once at least three English words are stable. It normally retranslates after at least 1.5 seconds and three more stable words, while queue coalescing removes obsolete revisions. The completed English wording refines the same translated cue. | Unit and broadcast tests prove unstable tails are excluded, a draft translates before final, cue identity is retained, and the final full cue remains eligible. Linux/native-reader validation remains required. |
| Retire unusably slow context batching | The operator no longer offers two-thought Contextual or four-thought Extended modes. Existing saved values migrate to Responsive Context in both runtime configuration and the live hub. | Configuration, UI-routing, and legacy-mode migration tests. |
| Prevent unexplained storage growth | Diagnostics now calculates application/data/model-cache storage on demand. Logs rotate at 5 MB with two backups, HTTP access logging is disabled, benchmark samples are memory-capped, and cleanup is allow-listed and operator-confirmed. | Storage, active-model protection, cleanup, rotation, and bounded-tail tests pass. |

## Required Linux Validation

Run A/A2 on this v0.7.0 release candidate using the same recording, audio route, Faster-Whisper model, device, and timing preset. If a historical comparison is needed, use the published v0.6.0 release. If English correction frequency and cadence pass, run Responsive Context, Live, and More Stable with Farsi, `zh-Hant`, and both languages on equivalent input at least three times.

Keep Responsive Context as the recommendation only when:

- English operational response has no material regression.
- `source_units.engine` is `word_timestamp_local_agreement_v5`, `interim_strategy` is `immediate_stable_prefix_guarded_edge_tail`, processing p95 is at most 0.010 seconds, and maximum is below 0.025 seconds.
- Faster-Whisper runs report `word_timestamp_passes` and `aligned_words`; the report must retain no words, times, or confidence values.
- When transcription compute p95 remains below the configured refresh interval, start-to-start `pass_interval` p95 should remain close to that interval rather than interval plus compute time.
- Edge-word withholding materially reduces visible last-word reversals without a material first-caption regression.
- First drafts, corrections, and extensions appear on their first recognition pass; there is no server revision hold or browser partial throttle.
- Cue lifetime p95 is materially lower than the previous run and `maximum_revision` no longer reaches long-lived values such as 54 during normal speech.
- Corrected A2 phrases revise one cue; legitimate spoken repetition remains visible.
- Every active language receives completed units in order.
- `cue_first_translation_publish` p50 is at most 3 seconds and p95 is at most 5 seconds on the Linux reference appliance, or any exception is explained by provider/model load.
- Translated draft revisions update one cue in place; completed translated cues do not change or reappear as duplicates.
- `source_units.final_units` is non-zero and boundary reasons look plausible.
- Any `final_jobs_superseded` event is explained by a newer revision of the same cue and the stale result is not published.
- Draft coalescing/backpressure explains overload without an accumulating delayed backlog.
- Each degraded event recovers after load falls.
- Native Farsi and Chinese reviewers confirm no new completeness regression.

The numeric report demonstrates timing and scheduler behavior. Human review remains required for meaning, naturalness, church terminology, script consistency, and reading comfort.
